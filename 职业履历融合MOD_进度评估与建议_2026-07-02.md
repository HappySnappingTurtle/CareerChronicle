# 职业履历融合 MOD（Career Chronicle）进度评估与建议

> 评估日期：2026-07-02
> 评估方式：独立客观视角——先核对设计文档声称的目标，再核对 `src/` 实际代码与数据，最后核对实机验证证据（日志、autotest 结果、截图）。**本文不以原设计文档为准绳，仅以"一个成熟 MC MOD 该是什么样"为准绳。**

---

## 一、总体结论（TL;DR）

**这个项目的真实状态比它自己的文档描述得更好。** 现状报告（06-04）和 Checklist 长期挂着"实机验收几乎为零"的警告，但 06-10 的自动化实机测试已经跑通了 122 步、19 项断言全部 PASS——包括最核心的"无序融合解锁"（blazing_charge / flame_arrow）、50 级 5 段进阶、死亡后数据持久化。日志显示 MOD 加载零报错，注册表 6 种族 / 11 职业 / 73 技能 / 26 融合全部就位，29 张有序截图覆盖了 GUI Scale 适配和死亡重生。

**核心判断：**

1. **玩法验证阶段已经完成**。产品的立身之本（无序履历池 → 融合解锁）在代码和实机层面都成立了。
2. **最大的架构债是技能效果不数据驱动**：73 个技能的实际效果全部硬编码在 1864 行的 `SkillExecutorRegistry` 里，数值写死。数据驱动框架是个"完整的外壳、硬编码的内核"。这在内容扩展期会成为主要瓶颈。
3. **最大的产品债是"隐藏职业的世界内线索"没有落地**：这是设计中最能出圈的卖点（不查 Wiki、靠古书/碑文/仪式解锁巫妖），目前只有 3 个 hidden_unlocks 的解锁条件判断，没有任何世界内引导内容。
4. **最大的观感债是程序生成的像素图标 + 零自定义音效**：140 张 PNG 全部由 Python 脚本绘制，67 处音效全部复用原版。对玩家而言这是"一眼 demo"的最直接来源。
5. **文档已经落后于代码**，多份 Checklist 之间状态互相矛盾，应该做一次归档合并。

**建议的下一步不是继续加内容，而是：清理文档 → 发布一个 0.3 公开测试版到 Modrinth → 用真实玩家反馈决定"技能系统重构"和"美术打磨"哪个先做。** 详见第五节。

## 三、三个视角的评估与建议

### 视角 1：MC MOD 开发者

**做得好的（不少地方超出个人 MOD 的平均水准）：**

- 服务端权威校验链完整且真实存在，C2S 包全部在主线程处理、取 `getSender()`，属性分配有白名单校验。这在职业/技能类 MOD 里是最容易偷懒出洞的地方，这里没有偷懒。
- Reload 原子性做对了：先解析到新 snapshot、校验通过才替换，失败保留旧注册表。这是很多知名数据驱动 MOD 都踩过的坑。
- Capability 带 `DATA_VERSION`、有死亡 `copyFrom`、NBT 键集中管理——为未来存档迁移留了口子。
- 双轨测试（GameTest 管注册表/公式/NBT 往返，autotest 管端到端流程 + 截图）是这个项目最值得保留的工程资产。
- S2C 特效包用 `PacketDistributor.NEAR` 32 格 + config 限流，有网络性能意识。

**问题（按严重度排序）：**

1. **`SkillExecutorRegistry` 1864 行是全项目最大的债**。73 个技能每个一段手写 Java，伤害 5.0F/6.5F、持续 tick、粒子数散落在方法体里。后果：① 加第 74 个技能的成本不降反升；② 数值平衡调整要重新编译，整合包作者无法调；③ 这个文件会一直膨胀到不可维护。设计文档承诺的"数据驱动、整合包可扩展"在技能效果层是落空的。
   **建议**：不必推倒重来。走"效果组件化"的渐进路线——把技能效果拆成可组合的原子操作（damage / apply_effect / knockback / heal / projectile / particle_ring / camera_shake），JSON 里写 `"effects": [{"op": "damage", "amount": 6.5}, ...]`，`SkillExecutorRegistry` 退化为组件解释器 + 少数真正特殊技能的手写 executor。先迁移最模板化的 40-50 个技能，特殊技能保留硬编码。这一步做完，"整合包作者加自定义技能"才真正成立。
2. **`src/test` 为空**。GameTest 需要拉起整个 MC 运行时，`CareerProgressionMath`、融合规则匹配、tag_scores 聚合这类纯函数完全可以用 JUnit 秒级验证。建议把纯逻辑测试下沉到 JUnit，GameTest 只留需要运行时的部分。
3. **datagen 未接入**：25 个 recipe、tags、绝大部分 model JSON 都是手写。1.20.1 的 `DataGenerator` 接入成本低，能消灭一整类"手写 JSON 拼错"的错误（你们的 `verify_resources.py` 本质上是在用脚本弥补 datagen 的缺位）。
4. **多人从未验证**。EULA 只是在 `run/eula.txt` 里写一行 `eula=true` 的事，不应该阻塞一个月。职业 MOD 的主战场是服务器，capability 同步、多人同时施放技能、玩家互相可见的特效，这些单人测不出来。
5. 卫生问题：`com.example.examplemod` 空壳目录未删；网络协议版本还是 `0.1.0`（内容已是 0.3 规模，协议变更后老客户端连新服务器的兼容策略需要想）；音效全用原版 `UI_BUTTON_CLICK` 这类占位。

### 视角 2：玩家

**会被打动的点：**

- "我是精灵出身、两段火法、一段弓箭的燃影魔箭手"这个幻想是真实成立的——履历池、无序融合、种族限制可选职业，机制都在。构筑型玩家（玩过 Origins/PMMO 觉得不够深的那批人）会认这个。
- 隐藏职业（巫妖/死亡骑士）+ 种族弱点（不死族怕神圣）这套暗黑幻想题材在 MC 职业 MOD 里差异化明显。
- 死亡不丢职业数据、防刷经验（刷怪塔收益衰减）这些设计对长线玩家友好。

**会劝退的点（按第一小时体验顺序）：**

1. **第一眼观感**：程序生成的 32×32 图标能看出"几何图形拼的"，技能音效全是原版 UI 音。玩家不会读你的架构，他们 5 分钟内就用图标和音效给 MOD 定档。这是目前"内在质量"与"外在感知质量"落差最大的地方。
2. **引导缺失**：一个新玩家如何知道"火法+弓手会解锁火焰箭"？如何知道巫妖存在？设计文档规划了世界内线索（古书/村民/碑文），但都没做。目前的手册是静态说明书，玩家最需要的是**融合预览**——在职业选择界面看到"选这个职业将解锁 ?/已知融合"，哪怕隐藏融合只显示剪影。没有这个，核心玩法对玩家是不可见的。
3. **26 个融合对 50 级重玩性的支撑存疑**：5 个职业段、8 个基础职业，理论组合空间远大于 26 个手写融合覆盖的范围。玩家第二个存档很快会发现"很多组合没有融合技能"，无序履历池的魅力会打折。这也回到开发者视角第 1 条——不解决技能数据驱动，融合数量就上不去。
4. **被动槽是空的、拖拽装配没有**：装配界面承诺了 4 被动槽但没有被动技能可装，属于"看得见的未完成"，比看不见的缺失更伤害信任。要么先藏掉，要么补上。
5. **多人不可用等于砍掉一半受众**：职业分工（我是牧师你是守卫）是职业 MOD 在服务器上的天然玩法，目前完全未验证。

### 视角 3：MOD 产品/创意

**差异化判断：成立，且这个空白至今仍在。** Origins 是"开局身份"、PMMO 是"熟练度数值"、各类职业包是"单线职业"——"无序履历池 + 重复计数专精 + 组合融合"确实没有现成竞品完整覆盖。机制本身（顺序不影响结果、次数影响倾向：fire2+bow1 偏爆发 vs fire1+bow2 偏精准）是一个能写进 MOD 页面第一段、让人看懂并想试的设计。这个核心值得坚持。

**产品层面的三个判断：**

1. **最有传播力的资产没有做**：隐藏职业的世界内解锁（在地牢里读到巫妖仪式的碑文、集齐材料做命匣）是最容易被 YouTube/B站 内容化的部分——"我在 MC 里解锁了巫妖"天然是一个视频标题。目前它只是 3 个后台条件判断。相反，投入很大的部分（742 行特效渲染、镜头震动）是玩家默认该有的。**创意资源的分配和传播价值是错位的。**
2. **应该尽早公开，而不是憋 1.0**：项目已经过了"闭门验证机制"的阶段，现在最缺的信息是"真实玩家会不会按你预期的方式构筑"。26 个融合够不够、专精缺失痛不痛、图标丑不丑到劝退——这些问题内部推演不出答案。Modrinth/CurseForge 发一个明确标注 Alpha 的版本 + 一个反馈 Discord/表单，比再开发一个月更有价值。MC MOD 社区对"机制新颖但美术粗糙的 Alpha"宽容度很高，对"憋了很久但玩法不对味的 1.0"宽容度很低。
3. **整合包生态是这个 MOD 的放大器，但门票还没买**：数据驱动注册层已经支持数据包扩展职业/技能，这是对整合包作者的核心卖点——但技能效果硬编码意味着他们只能重组你的技能，不能创造新技能；而且"数据包作者指南"一个字还没写。如果目标是被整合包收录（MC MOD 最大的分发渠道），效果组件化 + 扩展文档的优先级要提到美术之前。

**一个诚实的风险提示**：这类"构筑深度"MOD 的天花板取决于内容供给速度。Overlord 式幻想吸引来的玩家胃口很大，26 融合 + 3 隐藏职业大约支撑 10-20 小时新鲜感。如果后续内容全靠手写 Java，供给追不上消耗，口碑曲线会先扬后抑。技能组件化不只是技术债问题，是产品可持续性问题。

## 四、与原设计的分歧点（独立视角的不同意见）

- **原设计把"上限 100 级 / 10 段"列为后续大版本目标——建议放弃或无限期搁置。** 50 级 5 段已经需要 26+ 融合支撑，10 段的组合空间是指数级的，内容供给不可能跟上。深度应该来自"同样 5 段下更多职业和融合"，而不是更长的数轴。
- **原设计的"专精每职业 ≥4 层"在当前体量下过重。** 8 职业 × 4 层 = 32 个专精技能，全部要手写 executor。建议专精先做成"重复选择给已有技能升级"（数值/效果强化，组件化后即 JSON 改动）而非全新技能，成本降一个量级，玩家感知差别不大。
- **原设计按 0.1→0.2→0.3→1.0 闭门推进，验收标准是内部 Checklist。** 建议把"公开 Alpha 反馈"插入为 1.0 前的强制阶段——见下节。

## 五、建议的行动优先级

**P0 — 两周内（发布前置）**
1. 文档大扫除：归档 0.1 Checklist 和旧现状报告，只留一份反映真实状态的 STATUS.md（以 06-10 autotest 结果为基线）。
2. 确认 EULA，跑通 dedicated server + 2 客户端的 30 分钟多人测试（施放技能、互见特效、断线重连、数据持久化）。
3. 藏掉或补上被动槽；删除 `com.example.examplemod`；把网络协议版本与 MOD 版本对齐。
4. 写玩家指南（一页：怎么选种族→怎么升级→融合是什么），发 Modrinth Alpha + 反馈渠道。

**P1 — 一到两个月（由 Alpha 反馈校准顺序）**
5. 技能效果组件化（damage/effect/knockback/heal/projectile/particle 原子操作 + JSON 组合），迁移模板化技能，保留特殊技能硬编码。这是后续一切内容扩展的地基。
6. 融合预览 UI：职业选择时显示该选择将影响哪些融合（未解锁显示剪影）。这是让核心玩法"可见"的最小改动。
7. 隐藏职业世界内线索最小版：3 个隐藏职业各配 1 条战利品箱古书/成就链引导。
8. 纯逻辑下沉 JUnit；接入 datagen。

**P2 — 之后**
9. 图标手绘化或委托美术（可先只重绘 11 个职业 + 6 个种族图标，技能图标保留程序生成风格反而统一）；自定义音效替换最高频的施放/失败音。
10. 数据包作者指南 + 示例数据包（组件化完成后才有意义）。
11. 专精系统按"技能升级"方案落地；拖拽装配。
