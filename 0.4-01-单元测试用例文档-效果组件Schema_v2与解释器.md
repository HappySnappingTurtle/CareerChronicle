## 三、测试 Mock 策略

### 3.1 JUnit Mock 设计

- **MockCareerData**：实现 `ICareerData`，仅需 `getClassHistory()` 返回预设列表、`getUnlockedSkills()` 返回预设集合。其他方法可抛 `UnsupportedOperationException`。
- **MockRegistrySnapshot**：构造最小 `RegistrySnapshot`，仅填充测试需要的 ClassDef（含 tags 列表）。
- **CareerRegistry.snapshot() 替换**：JUnit 中需要一种方式注入测试用 snapshot。方案：`CareerRegistry` 添加 `static void setTestSnapshot(RegistrySnapshot)` + `static void clearTestSnapshot()` 方法（仅测试可用，通过包可见性或 @VisibleForTesting 限制）。每个测试的 `@AfterEach` 调用 `clearTestSnapshot()`。
- **EffectOpRegistry 清理**：每个涉及注册表的测试 `@BeforeEach` 重置（添加 `static void clearForTesting()` 方法）、`@AfterEach` 恢复。

### 3.2 GameTest Mock 设计

- GameTest 在 MC 运行时中运行，无需 mock。
- 使用 `helper.spawnWithNoFreeWill(EntityType.ZOMBIE, pos)` 生成目标。
- 技能释放通过 `CareerSkillService.useSkill(player, skillId)` 或直接调用 `EffectPipeline.execute()` 触发。

## 四、测试优先级

执行顺序（TDD：先写测试，后写实现）：

1. **P1 必须先行**：T-REG-*, T-PIPE-*, T-LVL-*, T-ORD-*（纯逻辑 JUnit，秒级反馈，覆盖核心正确性）
2. **P2 解析与校验**：T-PARSE-*, T-VAL-*（确保 JSON 格式正确解析）
3. **P3 下沉**：T-MATH-*（将现有 GameTest 的纯逻辑部分下沉为 JUnit）
4. **P4 运行时验证**：T-GT-*（GameTest，需 runGameTestServer，验证实际战斗行为）
